## Implementation summary

Per [§8.5 of the testing plan](../lighting-testing-plan.md#85-41--hdr-proxies). Replaces all 36 LED-related `PointLight`s with HDR-bright emissive proxies + halos and raises the `UnrealBloomPass.threshold` so only HDR-bright pixels (color × opacity > 1.0) bloom. The driver-write semantics for proxy + halo opacity are unchanged from baseline — what changes is the material color magnitude that the opacity multiplies into.

**Files changed (8 modified; 259 insertions, 172 deletions on branch `lighting/4.1-hdr-proxies` commit [`e0d413a`](https://github.com/ChessMess/UltimateDarkTowerDisplay/commit/e0d413a)):**

Multiplier applied to proxy + halo `color.r/g/b` so that `material.color × driver.v opacity` crosses the bloom threshold (1.0) at peak driver values.

Used everywhere a proxy/halo color is set or hot-reloaded.
- [`src/3d/LightingResolver.ts`](https://github.com/ChessMess/UltimateDarkTowerDisplay/blob/lighting/4.1-hdr-proxies/src/3d/LightingResolver.ts) — `DEFAULT_LIGHTING.scene.bloom.threshold` `0.0` → `1.0`. The single tuning knob that flips the bloom selector from "everything blooms" to "only HDR-bright pixels bloom" — this is the architectural change vs §4.5 (which kept threshold at 0.0).
- [`src/3d/SealManager.ts`](https://github.com/ChessMess/UltimateDarkTowerDisplay/blob/lighting/4.1-hdr-proxies/src/3d/SealManager.ts) — drop the 12 seal accent `PointLight` constructions in `buildSealBacklights`. `SealBacklightRef.light` becomes `THREE.PointLight | null` (always null in this branch). `buildSealBacklights` and `updateLighting` write the proxy + halo material color through `applyHdrColor`. `setSealLed`, `updateLighting`, `dispose` null-guard the `light` field.
- [`src/3d/Tower3DView.ts`](https://github.com/ChessMess/UltimateDarkTowerDisplay/blob/lighting/4.1-hdr-proxies/src/3d/Tower3DView.ts) — drop all 24 ring + ledge/base `PointLight` constructions in `buildLeds`. `LedRef.redLight` becomes `THREE.PointLight | null`. The bulk-lights gate machinery (`setBulkLightsVisible` / `updateLightsGate` / `prewarmLightPrograms`) survives with null-guarded writes, so the gate transition flag is still observable for tests and any future alternative that re-introduces per-LED PointLights can drop back in without re-wiring the path. `applyLightingToScene` HDR-scales ledge + base proxy + halo colors via `applyHdrColor`.
- [`src/3d/LedEffectAnimator.ts`](https://github.com/ChessMess/UltimateDarkTowerDisplay/blob/lighting/4.1-hdr-proxies/src/3d/LedEffectAnimator.ts) — `LedRef.redLight` nullable; `writeLed` null-guards `redLight.intensity`. Proxy + halo opacity writes are unchanged (still `driver.v` / `driver.v * halo.opacity`) — the HDR-scaled color is what newly carries them over the bloom threshold.
- [`tests/__mocks__/three.js`](https://github.com/ChessMess/UltimateDarkTowerDisplay/blob/lighting/4.1-hdr-proxies/tests/__mocks__/three.js) — added `Color.r/g/b/setRGB` so the HDR scale path is exercisable from unit tests.
- [`tests/unit/Tower3DView.test.ts`](https://github.com/ChessMess/UltimateDarkTowerDisplay/blob/lighting/4.1-hdr-proxies/tests/unit/Tower3DView.test.ts) — rewrote the 6 seal-backlight tests + 4 LED/gate tests to assert `light === null` / `redLight === null` and to drive proxy + halo opacity instead of PointLight intensity. New test verifies the HDR-scaled color path: `applyLightingConfig({ leds: { sealBacklights: { color: 0x00ff00 } } })` produces a proxy material whose `color.g ≈ 0xff/255 × 3.0`.

**Decision-on-scope — all 36 dropped (same as §4.5).** The literal text of [§4.1 in lighting-alternatives.md](../lighting-alternatives.md#41-hdr-color-proxies--raised-bloom-threshold-no-pointlights) names "all 36 PointLights" as the targets, and [§8.5's success criterion](../lighting-testing-plan.md#85-41--hdr-proxies) is "sequence `frameMs.median` at v-sync cap (≤ 16.7 ms) on Retina full-window" — both are unambiguous, so this implementation drops every LED-related `PointLight`: 12 seal accents (from `SealManager`), 12 ring inset reds + 12 ledge/base corner-near-surface reds (both from `Tower3DView.buildLeds`). Same scope as §4.5 — the §4.18 finding that the corner-positioned PointLights at `cornerNearSurfaceRadius=0.52` cast mostly outward (negligible interior spill contribution) still applies here.

**Tuning knobs chosen:**
- `HDR_PROXY_SCALE = 3.0` — starting value from the prompt's range. Visual eye-check at All-LEDs / Display canvas (see screenshot below) showed the bulbs reading as crisp bright red dots with bloom-amplified glow — comparable in pop to baseline. Did not tune further.
- `bloom.threshold = 1.0` — starting value from the §8.5 sketch. The `bloom.strength` (1.5) and `bloom.radius` (0.5) are unchanged from baseline per the no-other-config-changes rule.

The combination produces a peak rendered red of `linear(0xff2020).r × 3.0 = 1.0 × 3.0 = 3.0` for the seal/ledge/base proxies when `driver.v = 1.0` — well above the 1.0 bloom threshold. Half-brightness (`driver.v = 0.5`) lands the red channel at 1.5, still over threshold. Below `driver.v ≈ 0.33` the bloom contribution drops to zero — the LED becomes a non-bloomed dim red mesh until the driver climbs back. (For breathe/breatheFast/breathe50 effects this is the period right around the zero-crossing of the yoyo tween; at typical 4-second period this is a brief flash-off-then-on transition that reads as a "breathe through" rather than the smooth pulse the PointLight-driven version produced.)

No changes to `bloom.resolutionScale`, camera, DPR, audio, light counts in other places, or scene composition outside the named files.

## Interpretation

**Pass on perf — meets every gate from [§8.5's success criterion](../lighting-testing-plan.md#85-41--hdr-proxies). Mixed on visual — interior spill loss is significant and matches the documented §4.1 tradeoff exactly.**

- Sequence `frameMs.median` at Retina: **8.3 ms (v-sync cap)** — well under the ≤ 16.7 ms gate.
- Idle/empty regression: **none meaningful** — Display matches baseline exactly (8.3 ms), Retina is 8.4 vs 8.3 ms which is at the floor of measurement granularity.
- 1-LED and All-LEDs `frameMs.median`: **same as Empty** at both canvas sizes — the per-fragment cost is now flat regardless of LED state, exactly as the [perf skill's expected-signals table](../../.claude/skills/darktower-3d-perf/SKILL.md#when-evaluating-a-lighting-alternative) predicts for "remove PointLights" alternatives.
- Programs stable across 3 sequence-5 transitions: **yes** — 22 throughout, no recompile.
- Visual delta: **load-bearing tradeoff visible.** The HDR-scaled bulb proxies + halos at threshold 1.0 produce bloom-amplified glow that reads roughly as bright as baseline — see the 4 large bulbs at the bottom of the screenshot. But the drum-interior atmospheric spill (the warm red wash on drum surfaces that the 12 seal accent PointLights produced) is gone entirely. The seal cutouts read as small bright dots, not as glowing windows into a lit interior. Decision-relevance: §4.1 alone is unlikely to be acceptable as a shipping configuration; pairing with §4.4 (two-directional) or §4.5 (LightProbe) for spill recovery is the documented next step.

Compared to the prior Path A leaders:

|---|---:|---:|---:|---|

§4.1 matches §4.5 on perf to a noise-level tolerance — both alternatives drive every active-scenario window to the Empty v-sync ceiling. The decision-time tradeoff isn't about perf: §4.5 has a smooth global SH wash (no per-seal hot spots but the drum interior reads as lit), §4.1 has no spill at all (only bloomed bulbs/halos). On its own §4.1 is the more aesthetically dramatic departure — the drum interior reads black between the bulbs. Decision-relevance: §4.1 is a building block for the Path B combos (§4.4 / §4.19 / §4.11) rather than a shipping candidate on its own.

## Notable observations / risks

- **No ambiguity-rule recaptures needed.** Every active-scenario delta vs baseline was far outside the ±5% noise envelope (>87% improvement in every case). Empty matched baseline to within 1 absolute pp. Single-capture confidence is sufficient.
- **`programs` dropped from 30 (baseline) to 22 (this branch)** — same as §4.5 and not a regression. Removing the 36 PointLights eliminates the program variants keyed on `NUM_POINT_LIGHTS=36`. No new defines are added (the threshold change is a uniform, not a define), so the new stable count is 22 across all scenarios — the criterion that matters.
- **Interior spill loss is the load-bearing visual concern, exactly as flagged in [§4.1 of the alternatives doc](../lighting-alternatives.md#41-hdr-color-proxies--raised-bloom-threshold-no-pointlights) and [§8.5 of the testing plan](../lighting-testing-plan.md#85-41--hdr-proxies).** The screenshot pair makes this unambiguous — baseline shows a warm red wash on drum surfaces; §4.1 shows the surfaces dark with only the bulb + cutout positions glowing. The current implementation does not pair the drop with a spill recovery technique on purpose — this is the bake-off entry for "§4.1 alone." The next sessions in Path B (§4.4 two-directional, §4.19 interior-sprites, §4.11 min-cost-combo) chain on top of this baseline to recover spill at varying perf/aesthetic tradeoffs.
- **HDR proxy color is `linear(cfg.color) × 3.0` per the new `HDR_PROXY_SCALE` constant in `constants.ts`.** With `cfg.color = 0xff2020` (linear ≈ (1.0, 0.013, 0.013)) this produces a peak rendered red of ~3.0 in the red channel when `driver.v = 1.0`. The threshold (1.0) is reached when `driver.v ≈ 0.33`; below that the proxy doesn't bloom. For LEDs animating through breathe/breatheFast/breathe50 effects this is the period around the zero-crossing — a brief "off" between yoyo half-cycles. At baseline (PointLights animated by `intensity`) the same period was a smooth pulse-through-dim. Visually this reads as a slightly more "blinky" animation character on close inspection, but is invisible at typical viewing distance / canvas size.
- **`SpriteMaterial.color × AdditiveBlending` halos still bloom convincingly.** The halo sprites are also scaled by `HDR_PROXY_SCALE` — the additive blend math is `dst = dst + src.rgb * opacity`, so an HDR `src.rgb = (3, 0, 0)` with `opacity = 0.75` adds `(2.25, 0, 0)` to the destination, well above threshold. Halos still produce the diffuse glow read.
- **The bulk-lights gate machinery survives as dead weight.** `setBulkLightsVisible`, `updateLightsGate`, `lightsGateOpen`, and `prewarmLightPrograms` are kept (with null guards) so any future alternative that re-introduces per-LED PointLights can drop back into the same wire shape. Cleanup would be appropriate at decision time if §4.1 (or a §4.1-derived combo) wins.
- **three.js version anchor: `0.184.0`.** `UnrealBloomPass.threshold` semantics and `MeshBasicMaterial.toneMapped` have been API-stable since r100 (2018). `Color.setRGB` is core API. Risk profile here is lower than §4.2 (which patched shader chunk text); lower than §4.5 (which depends on `SphericalHarmonics3` ordering). The change is essentially "pick a different number" + "delete code."
- **The screenshot pair uses JPEG quality 85** rather than PNG, per the §4.5 session note. Both shots come in at ~795 KB each. PNG of the same captures would have been multi-MB.
- **No new unit tests were strictly required** since `LedEffectAnimator.writeLed`'s write semantics changed only via null-guards (existing tests exercise the proxy/halo opacity path). One new assertion was added for the HDR color hot-reload path (see Unit tests section).
- **Inheritability for chained alternatives.** §4.4 (two-directional) is the next Path B entry — it will add 2 `DirectionalLight`s back to the scene, expecting `LedRef.redLight = null` / `SealBacklightRef.light = null` to stay as on this branch. The seal-spill recovery via 2 globals plugs into the same HDR proxy + halo character §4.1 established. §4.19 (interior-sprites) similarly assumes no PointLights and adds additive sprites. §4.11 (min-cost-combo) additionally drops bloom. All three Path B sessions read §4.1 as the floor.
