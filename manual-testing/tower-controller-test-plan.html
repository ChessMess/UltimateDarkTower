What is the Tower Controller?

The Tower Controller is a reference control panel for the UltimateDarkTower library — a software driver for a motorized plastic tower called "Dark Tower" that ships with the board game Return to Dark Tower by Restoration Games.

The tower is a real device that connects to your computer or phone via Bluetooth. It has 24 colored LED lights (at different levels and positions), a speaker, and three rotating drums. The library lets you control all of these features from software.

This test plan walks you through every major feature of the Controller app — connecting, turning lights on and off, playing sounds, rotating the drums, managing game state (glyphs and seals), and monitoring the tower's health (battery, calibration status, dropped skulls). You don't need to know anything about the board game to run these tests.

Before You Start

Browser & Platform Support

This app uses Web Bluetooth to connect to the tower. Supported browsers: Chrome or Edge on desktop, or Chrome on Android. Not supported: Safari (Mac/iOS), Firefox, any browser on iOS (except third-party "Bluefy" app). This is a browser limitation, not the app's fault.

Choose Your Testing Track

🔴 Real Tower Track

You need: A physical Return to Dark Tower device, powered on, and within Bluetooth range.

Setup: Open the Controller app, leave the "Tower Type" dropdown set to BLE Tower (the default), then click Connect. You'll see your tower listed in the browser's Bluetooth device picker. Select it and pair. The connection banner will turn green: "Tower Connected".

Note: Some steps below are hardware-only (e.g., physically triggered skull drops). Those steps are marked [Hardware Only].

🟦 Tower Emulator Track

You need: Nothing but a modern browser. No physical device required.

Setup: Open the Controller app. Click the Tower Type dropdown and select Tower Emulator. A 3D-rendered tower will pop up in a new window. Click Connect in the main window. You'll connect instantly (no pairing needed).

Tip: This fully simulates the real tower's behavior. All controls work identically.

👉 Pro Tip: You can run both tracks at once — test once with the Emulator (quick feedback loop), then again with real hardware if you have it.

1. Connecting to the Tower

What this checks: The app can detect and pair with a tower (real or emulated), establish a Bluetooth connection, and show connection status.
1.1 Select tower type
Real Tower: Leave the "Tower Type" dropdown set to "BLE Tower" (the default). No action needed.
Emulator: Click the "Tower Type" dropdown at the top of the Controller page and select "Tower Emulator". A new window should pop up with a 3D tower model inside.
Expected: Emulator track — a popup appears titled "Tower Emulator" with a 3D tower visible. Real track — no visible change to the main app.
1.2 Click Connect
Real Tower: Click the green Connect button. A browser dialog will appear listing nearby Bluetooth devices. Find your tower (device name is "ReturnToDarkTower") and click it to pair.
Emulator: Click the green Connect button. Connection should happen instantly (no device picker).
Expected: The connection state banner at the top (near "Connect" button) turns green and displays "Tower Connected". The Disconnect button becomes enabled. Depending on device, you may see the 3D emulator tower light up or respond to show it's connected.
1.3 Verify connection state
Look at the right side of the page. You should see status cards for Battery (with a percentage), Calibration (with three status icons for Top/Middle/Bottom drums), and Skulls (a counter). The Calibration icons may show red X (not yet calibrated) or green check marks (calibrated).
Expected: Battery percentage appears (e.g., "85%"). Calibration icons are visible (likely showing red X for all three drums if this is the first connection). The Skulls counter shows "0" or some number. If any of these are missing or show "—" / "?", the connection is not working properly.
1.4 Cancel the Bluetooth device picker [Real Tower]
Real Tower: Click Connect to open the browser's Bluetooth device picker, then click Cancel (dismiss it without choosing a device).
Emulator: Not applicable — the emulator connects without a picker. Mark this step as the Real Tower track (or skip).
Expected: The app does not crash or hang. The connection banner stays on its previous text (it does not turn green or red). The only feedback is a red "Connection failed…" line in the Log panel on the right. (Note: a cancelled picker giving no banner feedback is a known rough edge worth flagging.)
1.5 Emulator popup lifecycle [Emulator]
Emulator: While connected via the Tower Emulator, close the 3D emulator popup window. Then reopen it by selecting "Tower Emulator" again from the Tower Type dropdown.
Expected: Closing the emulator popup automatically disconnects the session — the banner turns red "Tower Disconnected" within about half a second. (Also, if you are on a build without emulator support, selecting "Tower Emulator" shows an alert popup and reverts the dropdown back to "BLE Tower".)

2. Disconnecting

What this checks: The app can cleanly disconnect from the tower and show disconnected state.
2.1 Click Disconnect
Find the red Disconnect button (next to the Connect button) and click it.
Expected: The connection state banner turns red and displays "Tower Disconnected". The Connect button becomes enabled. The Disconnect button becomes disabled. Battery/Calibration/Skulls cards may show "—" or "?" to indicate unknown state.
2.2 Reconnect
Click Connect again. (Follow the same pairing flow as step 1.2 if using a real tower; Emulator reconnects instantly.)
Expected: Connection state banner turns green "Tower Connected" again. Status cards repopulate with battery/calibration/skulls.
2.3 Unexpected disconnect (no auto-reconnect)
Real Tower: While connected, power off the tower (or walk it out of Bluetooth range). Wait ~5 seconds.
Emulator: While connected, close the emulator popup window to simulate a dropped connection.
Expected: The banner turns red "Tower Disconnected". The three Calibration icons revert to gray/unknown. The Tower State byte packet blanks to zeros. The Battery card keeps showing its last value (it is not cleared). The app does NOT attempt to reconnect on its own — you must click Connect again. Confirm no auto-reconnect happens.
2.4 Controls while disconnected
While disconnected, try a few controls: click Play (Sound), toggle a light checkbox, click Rotate, click Calibrate, and the skull reset button.
Expected: Nothing is sent to a tower. Feedback is currently inconsistent by control: some do nothing silently (Calibrate, skull reset), some write a warning to the Log, and some log "Cannot send command: tower is not connected". Note which controls give no feedback at all — inconsistent disconnected-state handling is worth flagging.

3. Calibration

What this checks: The app can initiate a calibration routine, which teaches the tower the reference position of its three drums (required before rotation commands work reliably).
3.1 View calibration status
Look at the Calibration status card on the right side. It shows three small icons — one each for Top, Middle, and Bottom drums.
Expected: If not yet calibrated, all three show a red X (❌). If already calibrated, they show green check marks (✓). Hover over any icon to see a tooltip confirming "Top Drum", "Middle Drum", or "Bottom Drum".
3.2 Start calibration
Real Tower: Click the blue Calibrate button (in the Diagnostics tab). The tower will begin spinning its drums to find their zero positions. This takes about 10-15 seconds. Do not interrupt the tower during this time.
Emulator: Click the blue Calibrate button. In the emulator popup, the 3D tower will visually spin. Calibration completes almost instantly in the emulator.
Expected: A message appears near the Calibrate button saying "Calibrating...". The Calibrate button becomes disabled (grayed out). The 3D emulator (if using Emulator track) shows the drums spinning.
3.3 Verify calibration complete
Wait for the calibration to complete. The "Calibrating..." message should disappear, and the Calibrate button should become enabled again.
Expected: All three calibration icons turn green (✓). The message "Calibrating..." disappears. The Calibrate button is enabled again and clickable.

4. Lights

What this checks: The app can control the tower's 24 LEDs individually and in groups, apply different lighting effects (on, breathe, flicker), and turn all lights on or off.
4.1 Turn on a single light
Scroll down to the "Lights" section on the page. You'll see a grid of checkboxes labeled "Doorway" (12 lights: 4 sides × 3 levels), "Ledge" (4 corner lights), and "Base" (8 lights). Click on one of the checkboxes to select a light.
Expected: The checkbox is now checked. In the 3D emulator (if using Emulator track), that light LED should turn on. On a real tower, the physical LED lights up. The light state is reflected in the Tower State byte packet in the Diagnostics tab.
4.2 Turn off the light
Click the same checkbox again to uncheck it.
Expected: The checkbox unchecks. The LED turns off (in the emulator, the light dims or turns dark; on real hardware, the physical light turns off).
4.3 Turn on all lights
Above the light grids, find the blue All On button and click it.
Expected: All 24 light checkboxes become checked. On the real tower or in the emulator, all LEDs turn on (solid color, not blinking/breathing).
4.4 Apply a light effect
Before step 4.5, ensure all lights are on from step 4.3. Then, above the light grids, find the Light Style dropdown and select "breathe" (the options are code-style names: off, on, breathe, breatheFast, breathe50percent, flicker). Click the blue Apply to all lights button.
Expected: All lights begin pulsing/breathing (brightening and dimming in a cycle). In the emulator, you should visually see the lights pulse. On real hardware, the physical LEDs pulse.
4.5 Turn off all lights
Find the blue All Off button and click it.
Expected: All 24 light checkboxes become unchecked. All LEDs turn off (emulator or real tower).
4.6 Play a light override sequence
Below the light grids, find the Light Overrides control (a "select override" dropdown next to a Light Overrides button). Pick a sequence from the dropdown (e.g., "victory", "defeat", "dungeonIdle", "twinkle", "angryStrobe01") and click the Light Overrides button.
Expected: The tower/emulator runs the chosen built-in light animation (these are firmware sequences, different from the on/breathe/flicker styles). Each override should look distinct from the others. This is a separate feature from the "Apply to all lights" style controls above.
4.7 Lights turn off when switching tabs
Turn some lights on (e.g., "All On"). Then click a different tab in the left column (e.g., Library), then any tab other than Seals.
Expected: Switching tabs turns all lights off (the app calls "all lights off" on tab switch for every tab except the Seals tab). Confirm the lights go dark when you change tabs. This is intended behavior — verify it is consistent.

5. Sound & Volume

What this checks: The app can select and play sound effects from the tower's built-in sound library, and adjust the tower's speaker volume.
5.1 Select and play a sound
In the Diagnostics tab, find the Sound dropdown (it starts on "select sound"). Open it — it lists ~114 named sounds grouped by category (e.g., "Ashstrider", "Bane of Omens", "Gaze Eternal"). Pick any real sound, then click the blue Play button next to the dropdown.
Real Tower: The tower's speaker plays the selected sound.
Emulator: The sound plays from your computer's speakers (the emulator relays the audio).
Expected: A distinct, audible sound plays. The Log panel records an info line like "Playing sound N at volume V". If you click Play while the dropdown is still on "select sound", nothing plays and the Log shows "No sound selected".
5.2 Adjust volume (and check the arrow direction)
In the Diagnostics tab, find the volume control: two buttons "<" and ">" with a text level and a speaker icon between them. There are four levels: Loud → Medium → Quiet → Mute. Click each button a few times and watch the label and icon, then play a sound at each level.
Expected: The label steps through Loud / Medium / Quiet / Mute and the speaker icon changes to match; playback gets quieter (silent at Mute). Note the direction carefully: in the current build "<" moves toward Loud (louder) and ">" moves toward Mute (quieter). If that feels backwards from the "< = down / > = up" convention, flag it — the mapping is inverted. Buttons should stop (no-op) at Loud and at Mute.
5.3 No stop / no loop control
Play a long sound, then look for any way to stop it early or make it loop.
Expected: There is no "Stop" button and no loop toggle in the Controller — a sound plays to its end and cannot be interrupted from the UI (playback is always fire-once, non-looping). Confirm this is the case; note it as a coverage/UX gap rather than a defect.

6. Drum Rotation

What this checks: The app can rotate the tower's three drums to specific compass positions (North, South, East, West). The tower has three independent drums (Top, Middle, Bottom) that must all be calibrated before rotation works reliably.
6.1 Select positions and rotate the drums
In the Diagnostics tab, scroll to the "Drums / Rotation" section. Set the three dropdowns ("Top", "Middle", "Bottom") each to a different compass direction — North, South, East, West (e.g., Top = North, Middle = East, Bottom = South) — then click the blue Rotate button below them.
Real Tower: The drums spin to their targets over a few seconds; you may hear clicking/whirring from the motors.
Emulator: The 3D tower visibly spins its drums to the chosen directions.
Expected: The drums settle facing the chosen directions. The Tower State byte packet in the Diagnostics tab updates to show the new drum positions.
6.2 Rotation before calibration is allowed
Connect fresh (or reconnect) so the drums are not calibrated (Calibration icons show red X). Without calibrating first, set the drum dropdowns and click Rotate.
Expected: The Rotate command is not blocked — the tower still accepts and performs the rotation with no warning or error (unlike Move Glyph, which does require calibration — see the Glyphs section). Confirm rotation is not gated on calibration.

7. Randomizing Drums

What this checks: The app can rotate one or more drums to random compass directions in a single command.
7.1 Choose level(s) and randomize
Click the Library tab, scroll to the "Randomize Drums" section, and set the Level(s) dropdown (options: all, top, middle, bottom, top & middle, top & bottom, middle & bottom). Then click the blue Randomize button.
Expected: The selected drum(s) spin to random compass directions and the Tower State byte packet updates to the new positions. If you chose "all", all three drums move; if you chose a single level (e.g., "top"), only that drum moves while the others stay put.

8. Glyphs

What this checks: The app can track and manipulate glyphs — symbols drawn on the tower's drums that face different compass directions based on drum rotation. Glyphs represent in-game concepts (Cleanse, Quest, Battle, Banner, Reinforce).
8.1 View glyph positions
In the Library tab, scroll down to the "Glyphs" section. You'll see a 3×4 grid labeled with drum levels (Top / Mid / Bot rows) and compass directions (N / E / S / W columns). Below the grid is a dropdown labeled "Glyph:" to select a glyph (Cleanse, Quest, Battle, Banner, or Reinforce) and another labeled "Side:" to select a compass direction (North / East / South / West).
Expected: The grid displays. If any glyphs are currently active in their positions, those grid cells glow (amber/yellow highlight). Otherwise, all cells are dark/empty.
8.2 Move a glyph to a position
In the Glyphs section, select a glyph from the "Glyph:" dropdown (e.g., "Cleanse") and a side from the "Side:" dropdown (e.g., "North"). Then click the blue Move Glyph button. (This step assumes the drums are already calibrated — see 8.3 for the uncalibrated case.)
Expected: The tower's drums rotate so that the chosen glyph faces the chosen compass direction. The glyph-position grid updates to show the glyph now occupies that spot (one cell glows amber). The status display updates to show the new drum positions that put this glyph in place.
8.3 Move Glyph before calibration is blocked
Connect fresh so the drums are not calibrated (Calibration icons show red X). Without calibrating, pick a glyph and side and click Move Glyph.
Expected: The glyph does not move. The Log panel shows an error like "Unable to find current position for {glyph} glyph, please perform a calibration first." (Unlike drum Rotate, Move Glyph does require calibration because glyph positions are unknown until the drums are calibrated.)

9. Seals

What this checks: The app can track and manage seals at each of the 12 light positions (3 levels × 4 sides). Breaking a seal triggers a visual effect (lights flash) and updates app-tracked state; seals can also be cleared/reset. Note: seal state is software-only — tracked by the app, not the tower firmware — so it is lost on page reload. These controls are the Library tab's "Break Seal" fieldset. (A separate, emulator-only "Seals" tab manages 3D seal visibility — see section 16.)
9.1 View seal status grid
Scroll down to the "Seals" section in the Library tab. You'll see a 3×4 grid of square buttons (representing the 12 seal positions: 3 levels × 4 compass directions). A dropdown above the grid lets you select a seal location (e.g., "North Top", "South Mid", etc.).
Expected: The grid displays 12 empty (gray or white) squares. None are currently broken.
9.2 Break a seal via dropdown
Real Tower: Select a seal location from the dropdown (e.g., "North Top"). Click the blue Break Seal button. The tower will trigger a seal-break effect: lights at that position may flash, and a sound may play.
Emulator: Select a seal location from the dropdown. Click the blue Break Seal button. The emulator popup will show a visual seal-break effect (the seal at that position drops or disappears, and lights flash).
Expected: The corresponding grid square turns red/dark red (indicating the seal is now broken) and a seal-break effect fires on the tower/emulator. The Break Seal button is then disabled (greyed, ~50% opacity) for a 10-second cooldown (see 9.5).
9.3 Break a seal via grid click
Click directly on one of the gray (unbroken) seal squares in the grid (a square that is not red).
Expected: That square turns red (broken). A seal-break effect triggers on the tower/emulator. The dropdown updates to show the location you just clicked.
9.4 Clear all seals
Click the blue Clear Effect button (below the seal grid).
Expected: "Clear Effect" does two things: it turns off all lights and resets every broken seal. So all grid squares turn back to gray (not just the LEDs off), and the dropdown resets to "Select a Location". Verify that a previously-broken seal is restored, not merely un-lit.
9.5 Break Seal cooldown
Break a seal, then immediately try to break another one (via the button or by clicking a gray grid square) before ~10 seconds pass.
Expected: The second break is blocked during the 10-second cooldown — the Break Seal button stays disabled/greyed, and clicking a grid square logs "Break seal is on cooldown. Please wait before breaking another seal." After the cooldown the button re-enables and breaking works again.
9.6 Seal state persistence
Break one or two seals. Then (a) reload the browser page and reconnect; and separately (b) without reloading, switch the Tower Type between BLE Tower and Tower Emulator and reconnect; and (c) in the same mode, click Disconnect then Connect again.
Expected: Because seal state is software-only and in-memory: after a page reload (a) or a tower-type switch (b) the broken seals are lost (grid all gray). After a same-mode disconnect→reconnect (c) the broken seals are retained (the Tower object isn't recreated). Confirm each case.

10. Battery & Skulls

What this checks: The app displays the tower's current battery percentage and tracks the count of skulls (tokens) that have been dropped from the tower during gameplay.
10.1 View battery status
Look at the right side of the page. Find the Battery card (shows a battery icon). It displays a percentage (e.g., "85%") and a trend arrow (⬆️ increasing, ⬇️ decreasing, or ➡️ stable) showing whether battery level is rising, falling, or holding steady.
Expected: A battery percentage is visible. For Emulator, this is simulated and stable. For real tower, this reflects the actual tower's battery voltage.
10.2 View skull counter
Find the Skulls card on the right side (shows a skull icon 💀). It displays a number (e.g., "0") representing how many skull tokens have been dropped from the tower. Next to the number is a small icon button to reset the counter.
Expected: The skull counter is visible and displays a number (usually 0 if no skulls have dropped yet).
10.3 Reset skull counter [Hardware Only]
[Note: This step requires a real tower with skulls already dropped. Emulator does not simulate physical skull drops.] Click the small icon button next to the skull counter to reset it to 0.
Expected: The skull counter resets to 0. (No specific log line is emitted.) Note: the reset button only works while connected — clicking it after disconnecting does nothing.

11. Troubleshooting Modal

What this checks: The app can display a help/troubleshooting guide for common issues (tower jams, disconnects, firmware updates, battery specs).
11.1 Open the Troubleshooting modal
Scroll to the footer, find the Troubleshooting button, and click it.
Expected: A modal dialog opens overlaying the page, showing troubleshooting sections (e.g., "App Reports Tower Jam", "Tower Disconnects During Play", "Firmware Update Fails (Error 133 / Error 257)", "Battery Specifications") with readable instructions.
11.2 Close the modal
Close the modal by clicking the X button in the top-right corner, clicking outside the modal, or pressing Escape on your keyboard.
Expected: The modal closes and you're back at the Controller page.

12. Tower State Packet

What this checks: The Diagnostics tab shows the raw 20-byte status packet the tower reports, with per-byte tooltips, and keeps it in sync with commands and connection state.
12.1 View the Tower State packet
While connected, scroll to the Tower State fieldset in the Diagnostics tab. It shows 20 numbered bytes (Byte 0 … Byte 19). Hover over a byte to read its tooltip.
Expected: All 20 bytes are shown with values. Tooltips label them meaningfully (e.g., Byte 0 = Battery Level, Byte 19 = Checksum). Nothing shows "—" or blank while connected.
12.2 Packet updates on commands and blanks on disconnect
Rotate the drums (section 6) and watch the drum-position bytes. Then click Disconnect and look at the packet again.
Expected: The drum-position bytes change after a rotation. After disconnecting, the whole packet blanks to zeros.

13. Log Panel

What this checks: The always-visible Log panel (right column) can filter by level and by battery noise, filter by text, show a buffer counter, and copy/download/clear its contents.
13.1 Filter the log
In the Log panel, uncheck one of the level checkboxes (Info / Warn / Error / Debug). Set the battery filter radios (All / Changes / None). Type a word into the "Filter log messages..." box.
Expected: Unchecking a level hides those lines; the battery radios control how much battery-heartbeat noise is shown (None = hidden); the text box live-filters to matching lines only. Clearing the filters restores the full view.
13.2 Copy, download, and clear the log
Note the buffer counter (e.g., "12/1000"). Click Copy, then Download, then Clear Log.
Expected: Copy puts the displayed log on the clipboard; Download saves a "TowerLog.txt" file; Clear Log empties the panel and the buffer counter drops toward 0/1000. The counter tracks entries as new log lines arrive.

14. Charts — IR Detector Readings [Real Tower]

What this checks: The Charts tab plots the tower's differential IR detector readings over time, with a selectable window, start/stop collection, clear, and export. This data comes from real firmware, so the chart is essentially hardware-only.
14.1 Collect and display IR readings [Real Tower]
Real Tower: Open the Charts tab. Pick a time window (30 Seconds / 1 Minute / 5 Minutes), toggle the display checkboxes (IR Beam, Drum 1/2/3), and click Start. Interact with the tower (e.g., drop a token through the beam).
Emulator: Open the Charts tab and click Start — the chart is expected to stay empty because the emulator doesn't stream IR readings. Verify it simply stays empty (no error).
Expected: On real hardware the chart plots live values and the stats (Data Points, Latest/Min/Max) update; the status line reflects "collecting". The display checkboxes show/hide their series.
14.2 Stop, clear, and export
Click Stop (the Start button toggles), then Clear, then Export.
Expected: Stop halts collection; Clear empties the chart and resets the stats; Export downloads the collected data. (Collection also stops automatically on disconnect.)

15. BLE Debug — Flight Recorder

What this checks: The BLE Debug tab exposes the opt-in diagnostics "flight recorder": enable/capture toggles, live connection status and event stream, metrics, and a persistent incident log with export/clear.
15.1 Enable diagnostics (off by default)
Open the BLE Debug tab. Confirm "Enable BLE diagnostics (flight recorder)" starts unchecked. Check it, and optionally check "Capture command/response payload bytes".
Expected: Diagnostics are OFF until you enable them — this is a key precondition: if you don't enable it before a disconnect, nothing is recorded. Once enabled, the setting persists across a page reload (stored in localStorage).
15.2 Live status and event stream
With diagnostics enabled and the tower connected, watch the Live Status grid and Live Event Stream. Send some commands (rotate, play sound), then click Refresh.
Expected: Live Status shows Session ID, Connected = yes, ring-buffer fill, battery-sample count, last-heartbeat age, and GATT state. The event stream lists recent events (color-coded), including your commands. Refresh updates the view.
15.3 Capture and manage a disconnect incident
With diagnostics enabled, cause a disconnect (power off the tower, or close the emulator popup). Return to BLE Debug, click Refresh on the Incident Log. Try Export JSON on an incident and Export All JSON. Reload the page and check the incident is still listed. Then use Clear All.
Expected: A new incident appears with a cause and metrics. Export downloads JSON. Incidents persist across reload (stored in IndexedDB). Clear All prompts for confirmation and empties the log. (If you had NOT enabled diagnostics before the disconnect, no incident is recorded — verify that too.)

16. 3D Seals Tab [Emulator Only]

What this checks: The emulator-only "Seals" tab manages which of the 12 physical seal covers are shown on the 3D model. This is a different feature from the Library "Break Seal" section (section 9).
16.1 Tab shows a notice when not on the emulator
Real Tower / disconnected: Open the Seals tab (the 5th left-column tab) while connected to a BLE tower or while disconnected.
Expected: The tab shows a notice like "Connect with Tower Emulator to manage 3D seal visibility." The seal-toggle controls are not usable.
16.2 Toggle 3D seal visibility [Emulator]
Emulator: Connect via Tower Emulator, open the Seals tab. Click individual seal buttons (12 positions across Top / Mid / Btm), then use Remove All and Replace All.
Expected: Toggling a seal button shows/hides that seal on the 3D model. "Remove All" hides every seal; "Replace All" restores them. Confirm the 3D model reflects each change.
